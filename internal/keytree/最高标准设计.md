# keytree 最高标准设计


---


## 设计目标

| 目标 | 含义 |
|------|------|
| **路径即语义** | 所有运行时概念映射到 kvspace 路径，Unix /proc 哲学 |
| **纯字符串组合** | keytree 包不做 I/O、不调 kvspace、只组合路径字符串 |
| **所有权编码在路径中** | 路径前缀=所有权边界，谁拥有谁写入 |
| **禁止硬编码** | 所有路径只能经由 keytree 函数生成，grep hook 强制执行 |
| **`.` 引擎保留键** | kvlang 标识符不能以 . 开头，天然隔离引擎状态与用户变量 |
| **`[i,j]` 指令槽** | 函数体是二维表：行=指令序号，列=操作数角色 |

---

## 零、一句话定位

**keytree 是 kvlang 的路径系统**：将所有运行时概念映射到 kvspace 的键路径，
就像 Unix 将进程、设备、配置全部映射到文件系统路径一样。

keytree 包只做一件事：**把名字组合成路径**。没有 I/O，没有逻辑，只有字符串。

---

## 一、设计哲学（Unix 类比）

| Unix 文件系统 | kvlang keytree | 原则 |
|-------------|---------------|------|
| `/proc/<pid>/` | `/vthread/<vtid>/` | **所有权编码在路径中**：进程/虚线程的一切属性都在其路径下 |
| `/sys/class/net/eth0/` | `/sys/vm/<id>/` | **层次即结构**：路径结构反映事物的逻辑从属关系 |
| `/dev/tty0` | `/dev/tty/<name>/` | **设备独立命名空间** |
| `/lib/` | `/func/` | **可执行代码**：ELF 二进制 ↔ kvlang IR 文本，格式不同，语义相同 |
| `/proc/sys/kernel/pid_max` | `/vthread/seq` | **系统元数据是普通子项**，无特殊前缀 |

### 四条铁律

1. **`/` 是唯一分隔符**。路径中禁止出现 `:`。
2. **路径即拥有者**。X 的所有属性必须在 X 的路径前缀下，不得散落到其他根。
3. **一个节点可以同时是值和命名空间**。`/vthread/<vtid>` 存状态 JSON，同时有子项 `[0,0]`、`done`。
4. **键是纯绝对路径，各段只含 `[a-z0-9-]`**。路径段中不含 `.`；`.` 仅出现在槽的**值**中（如 `./slot` 相对引用），由执行引擎解释，与 keytree 无关。

## 二、路径全景树（filesystem view）

```
kvspace /
│
├── func/                              ← 可执行代码（类 /lib，IR 文本，与 /src/ 同格式不同阶段，经过编译优化）
│   ├── main                           ← 程序运行请求（JSON，VM 轮询消费）
│   ├── idx/
│   │   └── <funcname>                 ← 反向索引：函数名 → 包路径
│   └── <pkg>/
│       └── <name>                     ← 函数签名字符串（FuncSig.String()）
│           └── [i,j]                  ← 指令槽（i=行号，j=列：0=opcode，+n=write，-n=read）
│
├── src/                               ← 源码原文（ast写入，未经过编译优化）
│   └── <pkg>/
│       └── <name>                     ← 函数完整源码文本
│           └── [i,j]                  ← 指令槽（i=行号，j=列：0=opcode，+n=write，-n=read）
│
├── vthread/                           ← 虚线程运行时（类 /proc，高度动态）
│   ├── seq                            ← vtid 序列计数器（单 key，原子递增）
│   ├── ready                          ← 新 vthread 就绪通知队列（Notify/Watch）
│   └── <vtid>/                        ← vthread 根路径（DelR 整体清理）
│       ├── .pc                        ← 当前 PC（绝对路径字符串，必须以 /vthread/ 开头）
│       ├── .status                    ← 运行期 String: init|running|wait；
│       │                                终态 Del+Notify：值为 main() 返回值（如 "ok"/"error"）
│       ├── .<statusVal>/msg           ← 终态附加描述（正常运行时不存在）
│       │                                  .error/msg   — status="error" 时的错误详情
│       │                                  .timeout/msg — status="timeout" 时的超时说明
│       ├── [i,j]                      ← 当前帧指令槽
│       │   └── [i,j]/                 ← 子调用帧（嵌套 call 展开，理论上无限深度）
│       └── term                       ← 绑定终端名（可选，无则缺失）
│
├── dev/                               ← 设备（类 /dev）
│   └── tty/
│       └── <name>/
│           └── <stream>               ← stream ∈ {stdin, stdout, stderr, pty}
│
└── sys/                               ← 系统基础设施（类 /sys）
    ├── vm/
    │   └── <id>                       ← VM 实例状态 JSON：{status, pid, started_at}
    │       ├── hb                     ← 心跳 JSON：{ts, status, pid}
    │       └── cmd                    ← VM 命令队列（Watch/Notify）
    ├── op/
    │   └── <backend>/                 ← backend 名如 cuda、pytorch、jax
    │       ├── <n>                    ← 第 n 个实例状态（n = "0","1",…）
    │       │   └── cmd                ← 实例命令队列
    │       └── func/
    │           └── <name>             ← 算子函数定义

heap对象的路径，是自由的，交由kvlang的程序员自行配置路径，自行完成heap对象回收
```

## 三、指令槽约定 `[i,j]`

`[i,j]` 是矩阵坐标：`i` = 行（指令序号），`j` = 列（槽角色）。
函数体是一张二维表，行是指令，列是操作数：

```
 i \ j │ … │  -2  │  -1  │  0(op)  │  +1       │  +2  │ … │
───────┼───┼──────┼──────┼─────────┼───────────┼──────┼───┤
   0   │   │   2  │   1  │   add   │ ./result  │      │   │
   1   │   │      │   …  │  return │   …       │      │   │
```

| j | 语义 |
|---|------|
| `0` | opcode |
| `+1`, `+2`, … | 写槽（输出） |
| `-1`, `-2`, … | 读槽（输入） |

槽的**值**（参数名、立即数等）由执行引擎解释，keytree 不感知值的格式。

**链接帧（Link 模型）**：`call foo` 在 PC=`callPC`（如 `/_fn/[3,0]`）处展开：
- `callPC/_fn → /func/<pkg>/foo`（零拷贝，仅一条链接记录）
- `callPC/<param>` 存实参（帧局部变量，不经链接）
- `callPC/.callpc`, `callPC/.func`, `callPC/.ret0` 等引擎元数据

子帧路径 `callPC` = `parentFrameRoot/[3,0]`（`ChildFrameRoot` 推导），
路径深度 = 调用栈深度，无需额外栈结构。

## 四、路径规范（按根分组）

### 4.1 /func/

```
/func/main                  程序运行请求入口
/func/idx/<funcname>        反向索引：函数名 → 包路径
/func/<pkg>/<name>          函数签名字符串
/func/<pkg>/<name>/[i,j]    指令槽
```

### 4.2 /src/

```
/src/<pkg>/<name>           函数完整源码文本
/src/<pkg>/<name>/[i,j]     指令槽
```

### 4.3 /vthread/

```
/vthread/seq                vtid 序列计数器
/vthread/ready              新 vthread 就绪通知队列

/vthread/<vtid>/.pc             当前 PC（绝对路径，必须以 /vthread/ 开头）
/vthread/<vtid>/.status         运行期 String: init | running | wait
                                终态 Del+Notify：值为 main() 返回值（如 "ok"/"error"）
/vthread/<vtid>/.<statusVal>/msg 终态附加描述（正常运行时不存在）：
                                   .error/msg   → status="error" 时的错误详情
                                   .timeout/msg → status="timeout" 时的超时说明

/vthread/<vtid>/_fn             软链接 → /func/<pkg>/<name>（main 帧代码区）
/vthread/<vtid>/_fn/[i,j]      main 帧指令槽（读取时透明穿透链接）
/vthread/<vtid>/[coord]         子帧根（callPC = vthreadRoot/_fn/[coord]）
/vthread/<vtid>/[coord]/_fn     子帧链接 → /func/<pkg>/<name>
/vthread/<vtid>/[coord]/<param> 子帧局部变量 / 参数
/vthread/<vtid>/<...>/[coord]   可任意嵌套（路径深度 = 调用栈深度）
/vthread/<vtid>/term            绑定终端名（可选）

# 帧元数据（. 开头，引擎保留）
/vthread/<vtid>/[coord]/.callpc    callPC（供 HandleReturn 恢复）
/vthread/<vtid>/[coord]/.ret0     调用方写槽（return 时写入父帧）
/vthread/<vtid>/[coord]/.func     当前块函数名（TCO 每次更新，供调试）
/vthread/<vtid>/[coord]/.rootfunc 根函数名（CALL 时写入，TCO 不改变，供 resolveLabel 解析裸标签）
```

> `.` 开头的路径段是引擎保留键（类比 Linux 隐藏文件），kvlang 标识符不能以 `.` 开头，用户代码无法写入。
>
> **def main()->(.status:str)** — kvlang 约定：main 的返回变量名为 `.status`，
> 其终态值成为 `.status` 的 Notify 信号（如 `"ok"`、`"error"`）。
> WaitDone 的调用方收到该值并据此决定是否读取 `.<statusVal>/msg`。

### 4.4 /dev/

```
/dev/tty/<name>/<stream>    stream ∈ {stdin, stdout, stderr, pty}
```

### 4.5 /sys/

```
/sys/vm/<id>/hb             VM 心跳
/sys/vm/<id>/cmd            VM 命令队列
/sys/vm/<id>/err            VM 错误通知队列（unknown opcode 等）

/sys/op/<backend>/<n>       op 后端第 n 个实例状态
/sys/op/<backend>/<n>/cmd   实例命令队列
/sys/op/<backend>/func/<name>  算子函数定义
```

## 五、路径命名规范

| 规范 | 示例 | 禁止 |
|------|------|------|
| 用户路径段只含 `[a-z0-9-]` | `op-cuda`、`pre-main` | 大写、`_`、`:` |
| 路径分隔：只用 `/` | `/sys/vm/0` | `/sys/vm:0` |
| 路径不以 `/` 结尾 | `/vthread/42` | `/vthread/42/` |
| 数字后缀用裸数字 | `/sys/vm/0`、`/sys/op/cuda/1` | `/sys/vm/vm-0` |
| 保留段名 | `idx`、`seq`、`ready`、`term`、`hb`、`cmd`、`func` | 用户包/函数名避免与保留段名冲突 |
| 引擎保留键 | `.` 开头（如 `.pc`、`.status`、`.error/msg`） | 类比 Linux 隐藏文件；kvlang 标识符不能以 `.` 开头 |

## 六、禁止项

| 编号 | 禁止 | 理由 |
|------|------|------|
| F1 | keytree 包 import 任何 I/O 或业务包 | 它只组合字符串，永远不执行 |
| F2 | 路径中出现 `:` | Unix 路径无 `:`；与 IPv6 地址冲突 |
| F3 | 路径段中间出现 `.`（如 `foo.bar`） | `.` 在键中只允许出现在段的**开头**（引擎保留键），中间位置禁止；`./slot` 相对引用只出现在槽的**值**中 |
| F4 | 路径以 `/` 结尾 | 路径是标识符，不是目录字面量 |
| F5 | 在 keytree 之外硬编码根前缀字符串 | 所有路径只允许经由 keytree 函数生成 |
| F6 | 将同一概念的属性散布在多个根下 | 违反"所有权编码在路径中"原则 |
| F7 | 新增路径函数时不同步更新路径全景树 | 本文档是路径系统的唯一权威 |

## 七、路径所有权（并发写规则）

路径系统的核心承诺：**路径前缀即所有权边界**。
谁拥有某路径前缀，谁才有权写入其下的任意 key。

| 路径前缀 | 所有者 | 并发写规则 |
|---------|--------|-----------|
| `/vthread/<vtid>/` | 持有该 vthread 的单一 worker | **禁止**其他 worker 写入；热注入（调试/测试）除外 |
| `/func/` | 编译器 / 热替换工具 | **允许**随时更新；正在执行的帧已持有代码副本或 Link，不受影响 |
| `/src/` | 编译器 / 源码加载器 | **允许**随时更新；仅供工具读取，运行时不访问 |
| `/sys/vm/<id>/` | id 对应的 VM 实例 | 各 VM 只写自己的 `/sys/vm/<id>/`；跨实例只读 |
| `/sys/op/<backend>/` | op 后端服务 | 后端写自己的实例状态；kvcpu 只写 cmd 队列 |
| `/dev/tty/<name>/` | 设备驱动 | 设备层写入；用户 vthread 只读 stdin/stdout |

### 所有权三条推论

1. **写前缀自包含**：任何组件写入的路径，必须在其所有权前缀之内。
   违反此规则即出现"跨所有权写入"，是并发 bug 的根源。

2. **所有权可以是暂态的**：`/vthread/<vtid>/` 由 worker 在 pick 后独占，
   直到 vthread 终止（done/error）。GC 回收后，该前缀重新空闲。

3. **所有权与命名无关**：所有权边界由运行时行为决定，keytree 仅提供路径命名，
   不强制执行。kvcpu 的 `pick` 机制（原子写 status=running）是所有权的运行时实现。

## 八、完整路径示例（pre-main → calc → add，两次调用）

### 8.1 源代码（.kv）

```kv
def add(a:int, b:int) -> (c:int) {
    c <- a + b
}

def calc(x:int, y:int) -> (z:int) {
    add(x, y) -> z
}

calc(3, 4) -> result
```

### 8.2 加载后：/func/ 静态快照

`kvlang load` 将源码编译并写入 kvspace，路径永久不变：

```
/func/main                      = {"entry":"pre-main","reads":[],"writes":[]}


/func/pre-main             = "def pre-main() -> (result)"
/func/pre-main/[0,-2]      = 4
/func/pre-main/[0,-1]      = 3
/func/pre-main/[0,0]       = calc
/func/pre-main/[0,1]       = ./result
/func/pre-main/[1,0]       = return

/func/calc                 = "def calc(x:int, y:int) -> (z:int)"
/func/calc/[0,-2]          = ./y
/func/calc/[0,-1]          = ./x
/func/calc/[0,0]           = add
/func/calc/[0,1]           = ./z
/func/calc/[1,-1]          = ./z
/func/calc/[1,0]           = return

/func/add                  = "def add(a:int, b:int) -> (c:int)"
/func/add/[0,-2]           = ./b
/func/add/[0,-1]           = ./a
/func/add/[0,0]            = +
/func/add/[0,1]            = ./c
/func/add/[1,-1]           = ./c
/func/add/[1,0]            = return
```

### 8.3 执行期：/vthread/ 动态变化（Link 模型，/_fn 结构）

> 以 vtid=1 为例。serve.go 将入口调用指令写入 `[0,0]`，kvcpu 将其 rewrite 为 call，
> callPC = `/vthread/1/[0,0]`。

```
# ── T1：kvcpu 执行 [0,0]=pre-main → HandleCall ──────────────────
/vthread/1/.pc          = /vthread/1/[0,0]/_fn/[0,0]
/vthread/1/[0,0]/_fn    = Link:/func/pre-main   # 零拷贝挂载
/vthread/1/[0,0]/.callpc = /vthread/1/[0,0]
/vthread/1/[0,0]/.func   = pre-main

# ── T2：执行 pre-main/_fn/[0,0]=calc → HandleCall ────────────────
# callPC = /vthread/1/[0,0]/_fn/[0,0]
# ChildFrameRoot → /vthread/1/[0,0]/[0,0]
/vthread/1/[0,0]/[0,0]/_fn    = Link:/func/calc
/vthread/1/[0,0]/[0,0]/x      = 3
/vthread/1/[0,0]/[0,0]/y      = 4
/vthread/1/[0,0]/[0,0]/.callpc = /vthread/1/[0,0]/_fn/[0,0]
/vthread/1/[0,0]/[0,0]/.ret0   = ./z      # 调用方写槽

# ── T3：执行 calc/_fn/[0,0]=add → HandleCall（栈峰值）────────────
# callPC = /vthread/1/[0,0]/[0,0]/_fn/[0,0]
# ChildFrameRoot → /vthread/1/[0,0]/[0,0]/[0,0]
/vthread/1/[0,0]/[0,0]/[0,0]/_fn    = Link:/func/add
/vthread/1/[0,0]/[0,0]/[0,0]/a      = 3
/vthread/1/[0,0]/[0,0]/[0,0]/b      = 4
/vthread/1/[0,0]/[0,0]/[0,0]/.callpc = /vthread/1/[0,0]/[0,0]/_fn/[0,0]
/vthread/1/[0,0]/[0,0]/[0,0]/.ret0   = ./z

# 路径深度 = 调用栈深度（3 层 [i,j] 段 → 栈深 3）

# ── T4：执行 add/_fn/[0,0]= + ─────────────────────────────────────
# resolveReadValue(framePath=/vthread/1/[0,0]/[0,0]/[0,0], "./a") → "3"
/vthread/1/[0,0]/[0,0]/[0,0]/c = 7

# ── T5：add/_fn/[1,0]=return → HandleReturn ──────────────────────
# FrameRoot(/vthread/1/[0,0]/[0,0]/[0,0]/_fn/[1,0]) = /vthread/1/[0,0]/[0,0]/[0,0]
# retTarget = .ret0 = "./z" → 写 parentFrame/z
/vthread/1/[0,0]/[0,0]/z = 7
Unlink /vthread/1/[0,0]/[0,0]/[0,0]/_fn
DelR   /vthread/1/[0,0]/[0,0]/[0,0]

# ── T6：calc/_fn/[1,0]=return → HandleReturn ─────────────────────
# .ret0 = "./result" → 写 parentFrame([0,0])/result
/vthread/1/[0,0]/result = 7
Unlink /vthread/1/[0,0]/[0,0]/_fn
DelR   /vthread/1/[0,0]/[0,0]

# ── T7：pre-main/_fn/[1,0]=return（无 .ret0）→ HandleReturn ──────
# frameRoot=/vthread/1/[0,0] ≠ vthreadRoot → 嵌套 return
# nextPC = NextPC(/vthread/1/[0,0]) = /vthread/1/[1,0]
Unlink /vthread/1/[0,0]/_fn
DelR   /vthread/1/[0,0]

# ── T8：kvcpu 执行 /vthread/1/[1,0]，opcode="" → SetDone ─────────
/vthread/1/.status Del + Notify("ok")
```
