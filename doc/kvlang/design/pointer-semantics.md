# kvlang 指针语义设计分析：字符串 vs 指针的类型区分

> kvlang 中，"hello" 是字符串，"/vthread/1/data/0" 也是字符串。
> 若两者同为 `Value{kind:"string"}`，则 `*` 运算符无法区分"这是一个要打印的字符串"
> 还是"这是一个要解引用的路径指针"。

---

## 问题

```kvlang
"hello" -> _a          # _a = 普通字符串
"data/0" -> _ptr       # _ptr = 路径指针（也是字符串）
*_ptr -> v             # 解引用：kv.Get("data/0") → 正确
*_a -> v               # 解引用：kv.Get("hello") → 路径不存在，返回 nil
                       #   这是逻辑错误，但无法在编译期或运行期检测
```

根源：字符串和指针共享 `kind:"string"`，`*` 无法判断意图。

---

## 二、5 种参考语言如何处理

| 语言 | 字符串/地址是否区分 | 机制 |
|------|-------------------|------|
| **C** | ✅ 严格区分 | `char*` vs `int*` 都是指针，但类型系统区分。`"hello"` 是 `char[]`（数组），`&x` 返回 `int*`（指针）。不能拿字符串当指针用——`*"hello"` 是解引用字符串首地址，类型错误。 |
| **V8 (JS)** | ❌ 无指针概念 | 一切皆对象，引用是隐式的。`obj[key]` 动态属性访问。没有"取地址"或"解引用"，用方括号语法代替。 |
| **Go** | ✅ 严格区分 | `*int` 是指针类型，`string` 是值类型。`&x` 取地址，`*p` 解引用。字符串永远不是指针，`*"hello"` 编译错误。`unsafe.Pointer` 可绕过，但不鼓励。 |
| **Rust** | ✅ 严格区分 | `&T` 是引用（指针的 safe 版本），`String` 是拥有所有权的字符串。`*raw_ptr` 需 `unsafe` 块。引用和值有不同所有权语义，绝对不混。 |
| **Python** | ❌ 无指针概念 | 一切皆引用，但无显式解引用。`dict[key]` 动态键访问，`getattr(obj, name)` 动态属性。路径字符串如 `"/path/to/var"` 仍是普通字符串，由使用者决定是否当作路径解释。 |

### 行业共识

**静态语言（C/Go/Rust）**：字符串和指针严格类型区分。`*` 只接受指针类型。  
**动态语言（JS/Python）**：无指针概念，通过方括号/动态键访问代替。

kvlang 是**动态类型**语言（Value 带运行时 kind tag），但内存模型是**显式寻址**的（kvspace 路径即地址）。这使它处于一个独特位置：
- 像动态语言一样无编译期类型检查
- 像静态语言一样需要区分"值"和"值的地址"

## 三、四个方案

### 方案 A：新增 `ptr` kind（类型标记）

```go
// kvspace/value.go
func Ptr(path string) Value { return Value{kind: "ptr", raw: []byte(path)} }
```

| 变量 | 内部表示 | `*` 行为 |
|------|---------|---------|
| `_a = "hello"` | `{kind:"string"}` | ❌ 报错（不是 ptr） |
| `_ptr = data/0` | `{kind:"ptr"}` | ✅ kv.Get("data/0") |

**优点**：类型安全，`*` 只接受 ptr，拒绝 string。
**缺点**：新增 kind 常量，Value 序列化格式需支持 `ptr:` 前缀。

### 方案 B：`*` 不操作变量，操作内联路径表达式

```
*data[0]       → kv.Get(frameRoot/data/0)    路径内联，不经过中间变量
*data[_idx]    → kv.Get(frameRoot/data/<idx>) 动态下标
```

不存在"指针变量"这个概念——`*` 后面永远是一个路径构造表达式，
由编译器直接生成 kv.Get 调用。没有中间变量，就没有歧义。

**优点**：概念清晰，不引入新类型。`*` = "读这个路径"。
**缺点**：不支持"先把路径算出来存着，稍后再解引用"的间接模式。

### 方案 C：沿用当前 `kv.at` 风格，换名为 `*`

```
*data[0]       → 编译为 kv.at(data, 0)    语法糖，不变运行时
*data[_idx]    → 编译为 kv.at(data, _idx) 同上
```

`*` 是 `kv.at` 的语法糖——路径前缀和下标在语法层显式分离，
不产生"指针变量"。和方案 B 本质相同，但复用现有实现。

**优点**：改动最小（仅 parser + lowering），运行时零变化。
**缺点**：`*` 的语义被限定为"按前缀+下标读"，不是通用解引用。

### 方案 D：保持现状（不引入 `*`）

用户用 `kv.at(data, idx)` 读取路径，用 `kv.has(data, idx)` 判断存在。
for 遍历器内部使用这两个原语，对外暴露 `for (v in data) {}` 语法。

不对用户暴露指针概念。内部中间变量 `_N` 存的是路径字符串，
但这是编译器实现细节，用户不可见。

**优点**：零复杂度。用户不需要理解指针。
**缺点**：`kv.has`/`kv.at` 名字不直观，缺乏 C 语言直觉。

## 四、推荐方案

| | 复杂度 | 类型安全 | 对标语言 | 实现代价 |
|--|--------|---------|---------|---------|
| A (ptr kind) | 中 | ✅ | C/Go/Rust 的类型区分 | 新增 kind + 序列化 |
| B (内联路径) | 低 | ✅ | JS `obj[key]` | 新增 opcode |
| **C (语法糖)** | **最低** | ✅ | **Python `dict[key]` + C `*arr[i]`** | **仅 parser** |
| D (保持现状) | 零 | N/A | — | 零 |

**推荐方案 C**：`*data[idx]` 作为 `kv.at(data, idx)` 的语法糖。

对标 C 的 `*arr[i]`（解引用数组元素）和 Python 的 `d[key]`（动态键访问）。
kvlang 取两者之合：C 的 `*` 前缀表示"读这个地址"，Python 的方括号表示"动态下标"。

- `*` 不操作"指针变量"，操作"路径前缀+下标"对
- 下沉到 lower 时展开为 `kv.at` 调用
- 不存在"字符串 vs 指针"歧义——`*` 总是后跟路径构造语法
- 对标 C/Go/Rust：`*` 只接受"指针上下文"（此处是路径+下标），不接受裸字符串

```kvlang
# 用户代码
*data[0] -> v              # 读 data/0 的值
*data[_idx] -> v           # 读 data/<idx> 的值

# 编译器展开（lower）
kv.at(data, 0) -> v
kv.at(data, _idx) -> v
```

### 为什么不是方案 A（ptr kind）

方案 A 最"类型安全"，但 kvlang 是动态类型语言（对标 JS/Python），
引入 ptr 类型意味着用户需要理解"何时用 ptr、何时用 string"——
这更接近 C/Go/Rust 的思维模式，与 kvlang 的动态语言定位不一致。

### 为什么不是方案 B（内联路径）

方案 B 本质和方案 C 相同（`*` 操作路径表达式而非变量），区别是
是否需要新的运行时 opcode。方案 C 复用现有 `kv.at`，改动最小。

### 方案 C 的 Two Sum 验证

当前 kvlang Two Sum hash map：
```kvlang
set("/tmp", x, i+1) -> _       # "写入"路径 /tmp/<x>，存 i+1（避 0 falsy）
at("/tmp", need) -> j           # "读取"路径 /tmp/<need>
```

5 种语言的同段逻辑：

| 语言 | 写入 | 读取 |
|------|------|------|
| **C** | `ht[x] = i` | `j = ht[need]` |
| **Go** | `m[x] = i` | `j, _ = m[need]` |
| **Rust** | `map.insert(x, i)` | `map.get(&need)` |
| **Python** | `d[x] = i` | `j = d.get(need)` |
| **JS** | `m[x] = i` | `j = m[need]` |
| **kvlang 方案 C** | `i -> */tmp/x` | `*/tmp/need -> j` |

`*/tmp/x` 明确：读路径 `/tmp/<x>` 处的值。对标 C `*ptr`、Go `*ptr`、Rust `*ptr`。
`"/tmp"` 是字符串字面量，`*/tmp/x` 是路径解引用——`*` 消除歧义。

### 后续演进

若未来 kvlang 需要"先存路径、后解引用"的间接模式（如回调、虚表），
可在方案 C 基础上追加方案 A 的 `ptr` kind，两者不冲突：
- 直接访问：`*data[idx]`（方案 C，当前）
- 间接访问：`_ptr = &data[idx]; *_ptr`（方案 A，未来）
