# issue 分析：读写码约束下的设计取舍

> 读写码 = `writes <- opcode(reads)` 或 `opcode(reads) -> writes`。
> 每条指令显式命名所有读参和写参，无隐式栈，无匿名寄存器。
> 这是 kvlang 的最核心设计约束。

---

## 001: `/` 是浮点除法，无整数除法

| 维度 | 分析 |
|------|------|
| 是否为读写码问题 | ❌ 无关。整数除法是算子语义问题 |
| Go 怎么做 | `a / b` 整数相除得整数，浮点需 `float64(a) / float64(b)` |
| Python 怎么做 | `//` 整除，`/` 浮点 |
| **kvlang 建议** | 添加 `//` 整除算子（对标 Python），`/` 保持浮点 |

---

## 002: 禁止函数嵌套调用

| 维度 | 分析 |
|------|------|
| 是否为读写码问题 | ✅ **是读写码的设计约束** |
| 原因 | `int(n / 10)` 中 `n/10` 的"结果"没有具名写槽，违反了"每条指令的操作数必须是叶节点" |

这是读写码的正确约束。需要先写到中间槽：
```
n / 10 -> _tmp
int(_tmp) -> result
```

---

## 003: while 循环内无法提前退出 — ✅ 已修复

`break` / `continue` 已通过 `loopCtx` 实现，lower.go 在 while/for 中正确注入。

---

## 004: `x -> x` 是空操作

| 维度 | 分析 |
|------|------|
| 是否为读写码问题 | ❌ 不是。这是语法上的合法但无意义的指令 |
| 原因 | copy 指令做"读 x 的值，写到 x"，效果等于不执行。不是读写码的设计 bug，是用户误用 |

## 005: 数组只读，无法原地修改 — ⚠️ 待修复

| 维度 | 分析 |

| 是否为读写码问题 | 无关。这是缺少 `set` 算子 |

## 006: `print("str:", val)` 冒号解析 — ✅ 已修复

scanner 数字扫描已分阶段处理，不再把 `:` 当作数字字面量的一部分。

---

## 007: 无 `+=` 或 `++` 简写

| 维度 | 分析 |
|------|------|
| 是否为读写码问题 | ✅ **是读写码的设计选择** |
| 原因 | `+=` 隐含了"读 i、加 1、写回 i"，但读写码要求显式写出所有读参和写参 |

```
读写码正确写法：  i + 1 -> i     显式：读 i, 算 i+1, 写 i
C 风格简写：      i += 1          隐式：读和写是同一个变量
```

读写码宁愿冗长但要**完全透明**。对标 LLVM IR 的 `%next = add i64 %i, 1`，同样是显式命名中间值。

---

## 008: while 条件不支持复合表达式 — ✅ 已修复

`lower.go` 的 `evalCond` 已支持复合条件。嵌套表达式（如 `p * p <= n`）通过 lower 展开为：
```
p * p -> _tmp
_tmp <= n -> _cond
while (_cond) { ... }
```
