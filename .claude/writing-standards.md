# kvlang 文章规范

> 本文规定如何为 kvlang 撰写对外文章。不包含文章内容本身，只规定约束条件。

---

## 一、必须传达的核心概念

每篇文章至少覆盖以下 2 项：

| # | 概念 | 一句话定义 |
|---|------|-----------|
| C1 | **代码与数据同树** | 指令是 KV path，变量是 KV path，frame 是 KV path — 同一棵 KV 树（kvspace） |
| C2 | **路径深度即调用栈深度** | `/vthread/1/a/b/c` 的 `/` 层数就是当前调用栈深度，无需额外栈数据结构 |
| C3 | **PC 是路径字符串** | 跳转 = 字符串拼接，调用 = 子树复制，与传统 VM 的整数 PC 完全不同 |
| C4 | **透明执行** | 外部可以通过 `GET /vthread/{vid}/.pc` 读取运行中程序的任何状态，无需 gdb |
| C5 | **代码和数据对 agent 透明** | agent 用同一套 KV 接口读代码（`/func/`）和读运行态（`/vthread/`） |
| C6 | **崩溃可恢复** | 状态在 kvspace（持久化 KV 存储），进程崩溃后重启从上次 PC 继续，无需重放日志 |
| C7 | **kvspace 抽象** | 存储层通过统一接口抽象，后端可替换（Get/Set/Del/Watch/Notify/Link），不对特定存储耦合。未来规划 RDMA 后端 |

---

## 二、叙事结构约束

### 2.1 开篇

- 第一段必须包含一句定位陈述（基于 C1-C6 组合）
- 前三屏必须出现可运行的代码或 KV 路径示例
- 不使用疑问句开头（"你有没有想过..."）

### 2.2 技术解释

- 解释 kvlang 的设计时，必须有对比对象。对比对象限：JVM、CPython、WASM、gdb、Plan 9
- 不使用其他小众 PL 项目作为对比对象
- 技术解释中出现的路径必须是 kvlang 真实使用的路径（参考 keytree 包）

### 2.3 收尾

- 明确列出当前局限性
- 给出可运行的试用方式（one-liner）

## 三、作者背景的使用规范

| 可以用 | 不可用 |
|--------|--------|
| "新能源车厂 AI Infra，训练优化" | 暗示公司背书或有内部使用 |
| "业余 side project" | "我们在蔚来用 kvlang" |
| "来自前线看到的问题" | 透露公司内部技术细节 |

背景出现一次即可，通常在开篇或结尾。不在正文反复提及。

---

## 四、对不同受众的侧重点

| 受众 | 应强调 | 应省略 |
|------|--------|--------|
| PLT 研究者 | C1-C3（PC 是路径、path depth = stack depth）、Plan 9 谱系 | AI 训练场景 |
| AI/Agent 从业者 | C4-C6（透明执行、agent 可读写、崩溃恢复） | 编译器实现细节 |
| 系统工程师 | C1、C6（代码数据同树、持久化执行） | PLT 理论对比 |
| 中文社区 | C1-C4 从零讲起，不假设读者知道 PLT 概念 | Plan 9/Linda 等历史引用 |

## 五、禁止事项

| # | 禁止 | 原因 |
|---|------|------|
| B1 | "革命性""颠覆""下一代""重新定义" | 触发 HN/Reddit 免疫反应 |
| B2 | 暗示公司背书 | 个人项目，不给公司惹麻烦 |
| B3 | 贬低任何技术/项目 | 不专业 |
| B4 | 承诺未完成功能（如 GPU、分布式） | 透支信任 |
| B5 | 隐藏或淡化当前局限 | 诚实列出：依赖外部 KV 存储、单进程模型、无 GC、预发布阶段 |
| B6 | 无代码空谈概念 | 技术文章第一屏必须有路径或代码 |
| B7 | AI 生成的宣传腔 | 保持个人声音 |
| B8 | 中英文混用同一篇 | 各社区单独写 |
| B9 | 短时间内多社区同时发 | 间隔 ≥ 1 周，根据反馈迭代 |

---

## 六、发布流程

1. 确定目标社区和受众
2. 按 §一 选择本文覆盖的核心概念（≥2 项）
3. 按 §四 决定侧重点
4. 写好草稿 → 本地检查 §二 结构约束和 §五 禁止项
5. 发布
6. 收集反馈 → 决定下一篇方向

---

发布前逐项确认：

- [ ] 对比对象限知名技术

- [ ] 无 B1-B9 禁止项
