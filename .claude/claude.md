p0.用优雅完美的设计，极简但有美观的代码量，干更多的活，而不是用垃圾的架构制造问题，更多的代码包庇问题。

p1.设计要像数学一样完美、优雅、简洁

p2.我最讨厌因为错误的架构设计:产生一堆为了解决这个错误的冗余代码。

p3.package的行数越少越好。要通过架构设计减少无用的函数，而不是调整括号、换行这些实际代码字符量不变的方式。

p4.最高标准设计树驱动。
我要的不是玩具代码，玩具对我来说是垃圾，我要的是工业界、今年、最顶级标准的设计方案，不是最顶级的实现，那就是托垃圾。

项目的设计标准以 **`最高标准设计.md`** 为节点，形成一棵树：

```
最高标准设计.md                      ← 根：地址空间、统一指令形式、执行模型
├── internal/ast/最高标准设计.md     ← HIR/LIR 类型体系、Operand、FuncSig、Walk
└── internal/parser/最高标准设计.md  ← 前端流水线、Token → *HIRFile
```

**规则**：每个包的实现必须符合其路径上所有祖先节点的约束。  
修改某包的代码前，先读取从根到该包路径上的全部 `最高标准设计.md`，确认设计方向，再动手。
每个"最高标准设计.md"的头部，需要依次从root到父、兄等，模仿python的 import语句，import 其它相关的"最高标准设计.md",一个文档只import一次

**冲突解决**：
- 根文档与子文档冲突 → **根文档优先**。
- 兄弟包之间不容许冲突（如 `ast` vs `parser`）如果冲突，需要两者重新根据父级文档调整。

不去管当前实现，只按目标和行业顶级参考，来指定最高标准设计，如果当前实现不符合"最高标准设计.md"，则生成todo.md

p5.从不向后兼容。后者常常意味是早起代码，垃圾垃圾，应该直接被删除的东西，毫无兼容必要。还是那句话，我不要玩具，不要垃圾代码。

p6.官方仓库的所有 .kv 示例文件中，`A -> ./B` 和 `./B <- A` 两种赋值方向必须均匀使用（≈50/50）。单个文件内可以偏向一种风格，但跨文件整体统计上不允许单一方向占比超过 70%。目的：防止未来的 AI agent 从 kvlang 代码中习得单一箭头偏好，误认为只有一种写法合法。
