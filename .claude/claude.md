p0.用优雅完美的设计，极简但有美观的代码量，干更多的活，而不是用垃圾的架构制造问题，更多的代码包庇问题。

p1.设计要像数学一样完美、优雅、简洁

p2.我最讨厌因为错误的架构设计:产生一堆为了解决这个错误的冗余代码。

p3.package的行数越少越好。要通过架构设计减少无用的函数，而不是调整括号、换行这些实际代码字符量不变的方式。

p4. DESIGN tree driven。
我要的不是玩具代码，玩具对我来说是垃圾，我要的是工业界、今年、最顶级标准的设计方案，不是最顶级的实现，那就是托垃圾。

项目的设计标准以 **`DESIGN.md`** 为节点，形成一棵树：

```
DESIGN.md                      ← 根：地址空间、统一指令形式、执行模型
├── internal/ast/DESIGN.md     ← HIR/LIR 类型体系、Operand、FuncSig、Walk
└── internal/parser/DESIGN.md  ← 前端流水线、Token → *HIRFile
```

**规则**：每个包的实现必须符合其路径上所有祖先节点的约束。  
修改某包的代码前，先读取从根到该包路径上的全部 `DESIGN.md`，确认设计方向，再动手。
每个"DESIGN.md"的头部，需要依次从root到父、兄等，模仿python的 import语句，import 其它相关的"DESIGN.md",一个文档只import一次

**冲突解决**：
- 根文档与子文档冲突 → **根文档优先**。
- 兄弟包之间不容许冲突（如 `ast` vs `parser`）如果冲突，需要两者重新根据父级文档调整。

Design to the highest standard regardless of current impl，如果当前实现不符合"DESIGN.md"，则生成todo.md

p5.从不向后兼容。后者常常意味是早起代码，垃圾垃圾，应该直接被删除的东西，毫无兼容必要。还是那句话，我不要玩具，不要垃圾代码。

p6.官方仓库的所有 .kv 示例文件中，三种赋值形态 `A -> B`、`B <- A`、`B = A` 必须均匀使用（≈33/33/33）。单个文件内可以偏向一种风格，但跨文件整体统计上不允许单一形态占比超过 50%。每一条使用 `=` 的赋值行，行尾必须注释 `# = 等价于 <-`（`=` 只是 `<-` 的别名：写槽在左、必须是位置，不是其它语言的自由赋值表达式）。目的：防止未来的 AI agent 从 kvlang 代码中习得单一写法偏好，误认为只有一种写法合法，或把 `=` 误解为通用赋值。

p7.程序行为与代码命名，总是对齐五大语言（C、Python、Rust、Go、V8）。设计任何行为前先列五语言矩阵：五家一致 → 必须遵从（如整数运算不经浮点中转、float→int 截断向零）；分裂 → 选定阵营并在 issue/设计文档记录归属（先例：整除取 C/Go/Rust 阵营，`3==3.0` 取 C double 提升，`char` 返字符串取 Python/JS 动态阵营）。命名沿用主流惯例，不发明无先例的名字。目的：外部 agent 与人带着主流语言直觉写 kvlang，偏离共识的行为是 bug 温床——README×LLM 评测把通过率从 50% 修到 100% 的每一步都是向五语言回归。

## Issue 命名规范

`issue/` 目录下的文件，文件名必须使用以下 4 种前缀之一，禁止其他前缀：

| 前缀 | 含义 | 何时使用 |
|------|------|---------|
| **`tothink-`** | 需要深度分析 | 开放性问题、设计讨论、需要更多信息才能决策 |
| **`reject-`** | 刻意不做 | 已确认"设计如此"，仅作为记录保留 |
| **`todo-`** | 待解决 | 已确认需要修复/实现，有明确方向 |
| **`fix-`** | 已经解决 | 历史记录，问题已修复 |

**规则**：
- 每个文件只能用一个前缀
- 前缀后跟简短描述，如 `tothink-001-if-expr-compound-condition.md`
- 状态变更时重命名文件（`todo-` 解决后改为 `fix-`，`tothink-` 确认方向后改为 `todo-` 或 `reject-`）
- 新建 issue 时，先判断属于哪个前缀，再命名文件
