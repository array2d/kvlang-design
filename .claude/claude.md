# 我的原则

p0 绝对上帝设计美学：代码必须按我定义的清晰，优雅，完美，简洁的方式运行，非我的定义，直接error/panic/throw，不容许控制流包庇错误。

p1 绝不容许代码以奇怪的方式成功运行，就像用四个乳头走路的牛，必须立即杀死这样的牛。

p2 写代码要像疯子一样洁癖偏执，让代码走唯一的正确方向，其它方向直接error/panic，就像大树只容许主干向上生长，要砍掉所有分支，这些分支都是累赘！是bug的温床！。

p3.我最讨厌因为错误的架构设计:产生一堆为了解决这个错误的冗余代码。
 
p4. 最高工业界设计标准的驱动。
我要的不是玩具代码，玩具对我来说是垃圾，我要的是工业界、今年、最顶级标准的设计方案，不是最顶级的实现，那就是托垃圾。

p5.从不向后兼容。后者常常意味是早起代码，垃圾垃圾，应该直接被删除的东西，毫无兼容必要。还是那句话，我不要玩具，不要垃圾代码。

p6.官方仓库的所有 .kv 示例文件中，三种赋值形态 `A -> B`、`B <- A`、`B = A` 必须均匀使用（≈33/33/33）。单个文件内可以偏向一种风格，但跨文件整体统计上不允许单一形态占比超过 50%。每一条使用 `=` 的赋值行，行尾必须注释 `# = 等价于 <-`（`=` 只是 `<-` 的别名：写槽在左、必须是位置，不是其它语言的自由赋值表达式）。目的：防止未来的 AI agent 从 kvlang 代码中习得单一写法偏好，误认为只有一种写法合法，或把 `=` 误解为通用赋值。

p7.程序行为与代码命名，总是对齐五大语言（C、Python、Rust、Go、V8/TypeScript）。设计任何行为前先列五语言矩阵：五家一致 → 必须遵从（如整数运算不经浮点中转、float→int 截断向零）；分裂 → 选定阵营并在 issue/设计文档记录归属（先例：整除取 C/Go/Rust 阵营，`3==3.0` 取 C double 提升，`char` 返字符串取 Python/JS 动态阵营）。命名沿用主流惯例，不发明无先例的名字。目的：外部 agent 与人带着主流语言直觉写 kvlang，偏离共识的行为是 bug 温床——README×LLM 评测把通过率从 50% 修到 100% 的每一步都是向五语言回归。

p8-issue迭代

`issue/` 目录下的文件，文件名必须使用以下 4 种前缀之一，禁止其他前缀：

| 前缀 | 含义 | 何时使用 |
|------|------|---------|
| **`tothink-`** | 需要深度分析 | 开放性问题、设计讨论、需要更多信息才能决策 |
| **`reject-`** | 刻意不做 | 已确认"设计如此"，仅作为记录保留 |
| **`todo-`** | 待解决 | 已确认需要修复/实现，有明确方向 |
| **`fix-`** | 已经解决 | 历史记录，问题已修复 |

**规则**：
- 每个文件只能用一个前缀
- 前缀后跟简短描述，如 `tothink-001-if-expr-compound-condition.md`
- 状态变更时重命名文件（`todo-` 解决后改为 `fix-`，`tothink-` 确认方向后改为 `todo-` 或 `reject-`）
- 新建 issue 时，先判断属于哪个前缀，再命名文件
